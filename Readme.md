# 💸 PayFlow — Spring Boot Backend API

## 📌 Overview

PayFlow is a simplified backend system that simulates how digital payment platforms handle user registration and money transfers. It provides REST APIs for managing users and recording transactions, built using Spring Boot and an H2 in-memory database.

---

## 🚀 How to Run the Application

### Prerequisites

* Java 17+
* Maven installed

### Steps

1. Clone the repository or open the project in your IDE.
2. Navigate to the project root directory.
3. Build the project:

   ```
   mvn clean install
   ```
4. Run the application:

   ```
   mvn spring-boot:run
   ```
5. The application will start on:

   ```
   http://localhost:8080
   ```

## 🧱 Project Architecture

The project follows a layered architecture:

### 1. Controller Layer (`controller`)

* Handles HTTP requests and responses
* Maps endpoints like `/users` and `/transactions`
* Uses annotations like `@RestController`, `@RequestMapping`

👉 Example:

* `POST /users` → register a user
* `GET /users` → fetch all users

---

### 2. Service Layer (`service`)

* Contains business logic
* Acts as a bridge between controller and repository
* Ensures separation of concerns

👉 Example:

* Registering a user
* Sending money (saving transaction)

---

### 3. Repository Layer (`repository`)

* Handles database operations
* Uses Spring Data JPA (`JpaRepository`)
* No need to write SQL manually

👉 Example:

* `save()`, `findAll()`, `findById()`
* Custom query: `findByUpiId()`

---

### 4. Entity Layer (`entity`)

* Represents database tables
* Uses JPA annotations like `@Entity`, `@Id`, `@Column`

👉 Example:

* `User` → maps to `users` table
* `Transaction` → maps to `transactions` table

## Additional Questions Answers added

#### a. Paste SQL JPA generates for findByUpiId

```sql
select u1_0.user_id, u1_0.balance, u1_0.name, u1_0.phone_number, u1_0.upi_id
from users u1_0
where u1_0.upi_id=?
```

#### b.1 How JPA derives query

Spring Data JPA automatically generates SQL queries based on method names.
The method findByUpiId is parsed as:

findBy → indicates a SELECT query
UpiId → field name in the User entity

JPA converts this into a SQL query that searches the upi_id column in the database.


#### b.2 What ? means?

The ? in the SQL query is a placeholder used in prepared statements.
It is replaced at runtime with the actual value of upiId.
This prevents SQL injection and improves performance by reusing compiled queries.

#### c. Show the difference in what the User object looks like

with Request body 

```User{name='Ayan', upiId='ayan@okaxis', balance=1000.0, phoneNumber='9876543210'}```

without Request body 

```User{name='null', upiId='null', balance=0.0, phoneNumber='null'}```

When @RequestBody is used, Spring converts the incoming JSON payload into a Java object using Jackson. Without @RequestBody, Spring does not read the request body and instead tries to bind values from query parameters or form data. Since no such data is provided, all fields in the User object remain null.

## Custom Query Approaches in Spring Data JPA

Spring Data JPA provides three main ways to write queries:

### 1. Derived Query Methods

These are created automatically from method names.

Example:

```java
findByUpiId(String upiId)
```

Spring parses:

* `findBy` → SELECT query
* `UpiId` → field in entity

This generates SQL like:

```sql
SELECT * FROM users WHERE upi_id = ?
```

✔ Pros:

* No SQL needed
* Simple and readable

❌ Cons:

* Limited flexibility

---

### 2. @Query (JPQL)

Custom queries written using JPQL (Java Persistence Query Language).

Example:

```java
@Query("SELECT u FROM User u WHERE u.balance > :amount")
```

✔ Pros:

* More flexible than derived queries
* Uses entity names and fields
* Database-independent

❌ Cons:

* Slightly more complex

---

### 3. Native SQL Queries

Direct SQL queries written using database syntax.

Example:

```java
@Query(value = "SELECT * FROM users WHERE balance > ?", nativeQuery = true)
```

✔ Pros:

* Full control over SQL

❌ Cons (WHY LEAST PREFERRED):

* Database-dependent (not portable)
* Harder to maintain
* Breaks abstraction of JPA
* Loses benefit of entity mapping

## 4. Conceptual Write-Up

### 1. Request Lifecycle

When a `curl` request sends a `POST /users`, it first reaches the embedded Tomcat server, which forwards it to the **DispatcherServlet** (front controller of Spring MVC). The DispatcherServlet consults the **Handler Mapping** to find the correct controller method. It then uses a **Handler Adapter** to invoke the appropriate method in `UserController`. Before invocation, the request body is converted into a Java object. Finally, the `createUser` method executes and returns a response which is sent back as JSON.

---

### 2. Serialisation

Spring uses the **Jackson library** to convert JSON data into a Java object (deserialization). When a JSON payload like `{"name":"Priya","upiId":"priya@okaxis"}` is sent, Jackson maps it to the fields of the `User` class. If the JSON key is `"upi_id"` instead of `"upiId"`, the mapping fails because the field names do not match. As a result, the `upiId` field in the Java object will be `null`. This happens because Jackson relies on matching property names unless explicitly configured otherwise.

---

### 3. Spring Boot Features

Spring Boot provides three key features: **embedded server, auto-configuration, and production-ready defaults**. The embedded server is seen when Tomcat starts automatically without external setup. Auto-configuration configures components like JPA and H2 database based on dependencies in `pom.xml`. Production-ready defaults are visible in default configurations such as port 8080, JSON conversion, and logging. In PayFlow, all these features reduce manual configuration and speed up development.

---

### 4. Spring vs. Spring Boot

If plain Spring was used, we would have to manually configure the server, set up XML or Java-based configurations, and manage dependencies explicitly. We would also need to configure the database, DispatcherServlet, and component scanning manually. Spring Boot eliminates this by providing auto-configuration and embedded servers. It reduces boilerplate and allows the application to run with minimal setup. This makes development faster and less error-prone.

---

### 5. Stateless REST

Stateless means that each request is independent and the server does not store any client-specific session data. In PayFlow, the `POST /transactions` endpoint processes each request without remembering previous ones. This is important when running on multiple servers behind a load balancer, as any server can handle any request. There is no need to synchronize session data across servers. This improves scalability and reliability of the system.

---

### 6. Persistence

If transactions were stored in a Java `List`, all data would be lost when the server restarts. This is because memory is temporary and does not persist data. By using the H2 database, data is stored in a structured and persistent manner. Persistence ensures that transaction records remain available even after application restarts. This is critical for a payments system where losing transaction history is unacceptable.


### Conclusion

Derived queries are best for simple use cases, JPQL is preferred for flexibility, and native queries should be avoided unless absolutely necessary.
